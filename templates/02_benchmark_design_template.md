# 02_Benchmark用例设计

## Benchmark 设计原则

Benchmark 是上线前设计的一组固定测试用例。它不是随机问题列表，而是从 `01_评测方案.md` 中的评测维度反推得到的测试集。

- 每个核心评测维度至少对应一类用例，不能出现“定义了指标但没有用例”的情况。
- 用例要覆盖重要的正常、边界、风险和已知易失败场景。
- 每类用例都要说明测试目的、输入形态、预期行为和典型失败模式。
- 核心用例应保持稳定，便于跨版本比较。
- 上线后的代表性真实 Bad Case 应补充为回归用例。

`输入` 表示用户直接提出的任务，`上下文/素材` 表示完成任务所需的背景、文档、数据或附件。短且清晰的内容直接内联；长文本、多文件、需保留结构、二进制文件或需独立冻结的内容使用 `benchmark_materials/`。两个字段都可引用素材，素材包不是所有 Benchmark 的必选项。

## 输入与素材保真设计

从 `01_评测方案.md` 复制本轮输入保真契约，并逐项说明用例如何覆盖。不得因为现有适配器只接受文本，就把真实 PDF、图片、表格、代码仓库、工具状态或多轮对话静默改写成 JSON。适配器无法消费目标载体时，记录入口能力缺口并停止执行准备；资产设计可以保留正确载体。

## 每类用例设计

从 `01_评测方案.md` 的场景专属评测维度反推用例类别。以下结构应按评测维度重复填写，每一类用例针对一种主要能力或失败模式，避免所有 case 都同时测试所有维度。每类用例可以包含多个具体 case，具体输入和用例编号写入 `03_benchmark_cases.csv`。

### 用例类别一：`待命名`

- 对应评测维度：`必须与 01 中的维度名称一致`
- 测试目的：`说明这类用例要验证什么能力或暴露什么问题`
- 输入形态：`用户输入，以及所需上下文、文档、图片、工具状态或其他素材`
- 用例设计方向：
  - `方向一：典型任务或常见输入`
  - `方向二：容易暴露该维度问题的变化输入`
- 预期行为：`说明合格输出必须满足什么，不要求写成唯一标准答案`
- 典型失败模式：
  - `失败模式一`
  - `失败模式二`
- 覆盖场景：`正常 / 边界 / 风险 / 已知易失败`
- 优先级：`P0 / P1 / P2`

## Benchmark 维护规则

- 保持核心用例稳定，便于版本对比。
- 新增用例需记录来源和原因。
- 删除或修改用例需记录影响范围。
- 对重大产品变更，应更新 Benchmark 覆盖范围。
- 不要把一次性噪声样本全部塞入 Benchmark，应优先沉淀代表性失败模式。
- 生成素材包时同步维护 `benchmark_materials/manifest.json`，CSV、Manifest 与全部引用素材作为一个 Bundle 冻结。
- 候选产品只接收解析后的输入和上下文；预期行为、失败模式、Rubric 和风险规则不得进入候选载荷。
- 文本素材可由 Runner 展开；二进制素材必须交给产品适配器通过真实入口处理。
- 内容完整与文件形态真实分别检查。不得把扩充文本称为完整文档，也不得把合成误写成压缩；合成表示虚构数据。
- 每次生成后按真实用户手动任务逐槽位核对输入，发现格式或链路不一致时披露并阻止端到端表述。

## Benchmark 与 Bad Case 的关系

`03_benchmark_cases.csv` 是稳定的 Benchmark 用例主表，不保存候选输出、Judge 结果或人工裁决。每轮运行通过 `runs/{run-id}/results.csv` 记录执行和 Judge 结果，必要时通过 `human_review.csv` 记录人工裁决。`07_bad_case_log.csv` 是长期失败案例台账。不是所有 Benchmark 失败都必须进入台账；应进入台账的是高严重度、触发一票否决、反复出现、线上真实反馈、影响核心任务或代表一类典型失败模式的问题。台账中的代表性案例应转化为 Benchmark 回归用例。

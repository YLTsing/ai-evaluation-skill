# 上线前离线评测流程

## 适用场景

用于 AI 产品进入灰度或上线前，验证模型、Prompt、RAG、Agent 或工作流版本是否满足准入标准。

## 流程

1. 读取用户材料，区分已确认产品事实、规范性要求和未知项。
2. 向用户确认实际用户、任务、输入、输出、工作流、RAG/模型配置、人工决策边界和本次评测目标。
3. 产品事实未确认时提问并停止，不创建输出目录或评测文件。
4. 产品事实确认后，定义评测目标、范围、成功标准和风险边界。
5. 设计指标：从通用维度出发，结合产品任务、失败模式和风险扩展为唯一一组场景专属评测维度；另行定义风险指标和一票否决项。
6. 设计 Benchmark：从每个场景专属评测维度反推用例类别，并覆盖正常、边界、风险、已知易失败和回归场景。
7. 编写 Rubric：沿用同一组维度名称，逐维度定义 3 分、2 分、1 分、具体示例和评分边界。
8. 生成 LLM-as-Judge 评审方案：明确输入字段、输出 JSON、回填规则和人工复核条件。
9. 若用户未说明是否执行，只询问一次“立即执行 / 仅准备执行包 / 暂不执行”；明确只要资产时不重复询问。
10. 用户选择执行准备时初始化 `run.json` 和预填充的 `results.csv`；暂不执行时不创建 `runs/`。
11. 用户选择立即执行时，由 Evaluation Runner 调用产品真实入口；不要用当前 Agent 代替产品，也不要用模拟输出代替真实评测。
12. 候选执行完成后检查成功、失败、空输出和缺失运行指标，只将成功且非空的输出交给 Judge。
13. 由当前调用本 Skill 的 Agent 结合 Rubric 执行 Judge，并将结构化结果回填 `results.csv`。
14. 有复核任务时生成 `human_review.csv`，对强制复核集合和普通样本分层抽样集合进行人工复核。
15. 检查报告门禁：强制复核和抽样复核完成率均为 100%，未决否决候选为 0。
16. 输出评测结果：门禁满足时生成正式报告；已有 Judge 结果但门禁未满足时生成初步分析；没有 Judge 结果时只保留运行记录。
17. 将值得沉淀的失败案例写入 Bad Case 台账。

## 输出

- 稳定的 `03_benchmark_cases.csv`
- 用户选择执行时生成的 `runs/{run-id}/run.json` 与 `results.csv`
- 存在复核任务时生成的 `human_review.csv`
- 按数据状态生成 `06_初步评测分析.md` 或 `06_评测报告.md`
- 更新后的 `07_bad_case_log.csv`

## 注意

没有真实模型输出、评分和必要人工复核时，不得生成正式上线结论。

# AI Evaluation Skill

## 1. 目标与边界

为 AI 产品生成结构化、可执行、可复用的评测资产，并指导用户用真实候选输出、LLM-as-Judge 结果、人工复核和 Bad Case 完成可信评测。不要把本 Skill 简化为通用打分器，也不要把资产设计、执行、报告、Bad Case 和 A/B 测试当作默认连续执行的单一路径。

本 Skill 支持评测资产设计、执行准备、真实候选执行、Judge 执行、正式评测报告、Bad Case 闭环、A/B 测试设计和 A/B 测试分析。Evaluation Runner 调用产品已经确认的真实执行入口；产品内部可以使用任意模型 API、本地模型、工作流、工具或混合架构。Judge Runner 由当前调用本 Skill 的 Agent 执行评审，除非用户明确配置其他 Judge 方式。

## 2. 不可越过的原则

- 先识别产品场景、用户任务、输入输出和风险，再设计指标与资产。
- 将用户确认的产品事实与知识库、PRD、Prompt 中的规范性要求分开；“产品应当怎样”不能证明“当前产品已经怎样实现”。
- 使指标可观察、可判断或可打分；不要直接套用脱离任务的通用指标。
- 使 Benchmark 覆盖正常、边界、风险和已知易失败场景，并保持稳定以支持跨版本比较。
- 将 `03_benchmark_cases.csv` 作为稳定的 Benchmark 用例主表。每轮候选执行和 Judge 结果保存在 `runs/{run-id}/results.csv`，人工裁决保存在按需生成的 `human_review.csv`；通过“评测运行 ID + 用例编号 + 运行序号”关联。
- 使用 Rubric 降低评分自由度，写清分档证据、冲突处理和适用边界。
- 将一票否决项与普通评分维度分开；平均分不得覆盖事实错误、公平性、安全、合规等否决项。
- Judge 只能标记“一票否决候选”。高风险场景的最终否决结论必须经过人工复核。
- 对高风险、异常分、否决候选执行强制人工复核，并按预先定义的规则抽样复核普通样本。
- 没有真实候选输出、评分、必要人工复核或 A/B 数据时，不得虚构正式分数、Bad Case、实验结果或决策结论。
- 只有用户明确要求“模拟”“示例”或“演示”时才生成模拟数据，并显著标注“模拟示例，非真实评测结果”。模拟数据不得作为上线依据。
- 将值得沉淀的 Bad Case 纳入分类、归因、修复、复测和 Benchmark 回流闭环。
- 使持续评测同时覆盖模型、Prompt、检索、工具、策略或交互变更触发，以及定期触发。
- 仅按第 8 节“统一输出契约”决定文件名和生成条件；其他章节不得另行放宽门禁。

## 3. 请求路由

先判断请求属于哪一种模式，再检查其最低输入。不要默认生成完整资产包或顺序执行后续阶段。

| 模式 | 典型请求 | 最低输入 | 执行动作 |
|---|---|---|---|
| 评测资产设计 | 明确要求“完整评测资产包”“全套评测资产” | 已通过第 4.1 节产品事实确认门禁 | 生成默认资产包，不生成结果或结论 |
| 单项资产生成 | 设计评测方案；只要 Benchmark、Rubric 或 Judge 方案 | 已通过第 4.1 节产品事实确认门禁及该资产的最小输入 | 只生成点名资产，补充边界和上下游依赖 |
| 执行准备 | 准备评测运行；生成待执行数据包 | Benchmark 已冻结；产品版本和计划运行次数明确 | 初始化 `run.json` 和预填充的 `results.csv`，不调用产品 |
| 候选执行 | 执行评测；运行 Benchmark；比较产品版本 | 执行准备完成；真实产品入口已配置并通过检查 | Evaluation Runner 调用产品入口，记录真实输出和运行指标 |
| LLM-as-Judge 执行 | 按 Rubric 评价候选输出 | 成功的真实候选输出、Rubric、Judge 方案 | 当前调用本 Skill 的 Agent 逐条评审并回填 `results.csv`，生成必要复核任务 |
| 正式评测报告 | 根据本轮结果出正式报告 | 候选输出、Judge 结果、规定范围的人工复核 | 先检查报告门禁；满足后生成正式报告，否则降级为初步分析 |
| Bad Case 闭环 | 分析失败案例或做阶段复盘 | 真实 Bad Case 或完整失败结果 | 更新台账，按需生成阶段分析、根因和回归建议 |
| A/B 测试设计 | 设计 A/B 测试 | 改动、组别、假设、指标等必要信息 | 只进入 A/B 设计分支 |
| A/B 测试分析 | 根据 A/B 数据判断 | 真实实验数据、组别、样本、周期、指标和风险事件 | 校验完整性后进入 A/B 分析分支 |

概念解释、方法讨论、现有内容评审和头脑风暴属于非产物型请求，除非用户明确要求，否则不要生成文件。

## 4. 数据状态与澄清

执行前先识别产品事实是否已确认，再识别当前数据状态：

1. 只有产品场景。
2. 已有 Benchmark，但尚未初始化评测运行。
3. 已准备评测运行，但尚未执行或执行未完成。
4. 已有成功的真实候选输出，但没有 Judge 结果。
5. 已有 Judge 结果，但人工复核门禁未满足。
6. 强制复核项和抽样复核均已完成。
7. 已有真实 Bad Case。
8. 已有真实 A/B 实验数据。

### 4.1 产品事实确认门禁

评测资产设计以及依赖产品事实的单项资产生成，必须先完成本门禁。不要把用户提出“帮我设计”视为同意按假设继续。

先整理一份“产品事实摘要”，逐项标记信息来源为“用户已确认 / 用户材料中的规范要求 / 尚未确认”：

1. 产品服务谁，在什么业务环节使用。
2. 产品实际帮助用户完成什么任务，哪些任务明确不做。
3. 产品实际接收哪些输入、字段和文件格式，哪些是必填项。
   对每个输入槽位分别确认用户实际操作、原始载体、产品接收边界和解析组件；“最终传给模型的是文本”不能替代“用户上传的是 PDF”。
4. 产品实际输出什么结构、结论类型和可执行动作。
5. 当前工作流、关键节点、条件分支、多轮状态和异常处理方式。
6. RAG、模型、Prompt、工具及其版本和实际配置。
7. 人工复核发生在哪一步，AI 输出如何影响最终业务决策。
8. 本次评测目的、范围、目标版本、产品真实执行入口和已有真实数据。

用户提供的知识库、PRD、Prompt、业务规则或示例只能作为规范性要求来源，不能据此推断当前产品的实际输入、输出、工作流节点、模型配置、检索配置、引用能力或上线状态。

若存在会影响指标、Benchmark、Rubric、Judge 或风险规则的未确认项，必须：

1. 从第 4.2 节选择 5–8 个必要问题，以对话方式提问。
2. 提问后停止，不创建产品输出目录，不生成或更新任何评测文件。
3. 等待用户确认产品事实摘要，或明确回复“按假设生成草案”。
4. 只有门禁通过后才能进入第 5.1 节的资产设计步骤。

如果用户明确要求“按假设生成草案”，保持稳定文件名，但在文档开头标记“草案，包含未经确认假设，不可直接用于执行或决策”，并逐条列出假设。用户已在当前对话明确确认完整事实，或明确要求“无需再次确认，按以下事实生成”时，可视为门禁通过。复用已有产品目录时，若产品事实或范围发生变化，只确认变化项。

### 4.2 澄清问题

优先从以下问题中选择 5–8 个必要问题，不要机械照抄清单：

- 产品帮助谁完成什么任务？用户输入和产品输出分别是什么？
- 它属于 RAG、Agent、写作、审核筛选、客服、数据分析、代码助手、多模态还是其他类型？
- 哪些错误会伤害用户或业务？是否涉及安全、公平、合规、隐私、财务、医疗、法律或品牌风险？
- 本次目标是资产设计、上线前准入、版本对比、上线后监控、Judge 执行、Bad Case 闭环还是 A/B 测试？
- 是否已有真实案例、Benchmark、候选输出、Judge 结果、人工复核、历史 Bad Case 或业务指标？
- 用户要单项资产还是完整资产包？是否要求模板？
- 若需要执行评测，产品可通过什么真实入口调用：安全的命令、HTTP 接口，还是导入已有结果？

不得自行用假设补齐产品事实。即使用户允许生成草案，也不得补造候选输出、评分、人工复核、Bad Case 或实验数据。

A/B 请求还应确认改动点、对照组、实验组、分流单位与随机化方式、核心假设、主指标、护栏指标、实验周期、样本量或停止规则、离线准入结果，以及是否已有真实实验数据。

## 5. 主流程与数据门禁

### 5.1 评测主流程

1. 路由请求，区分单项资产与用户明确要求的完整资产包。
2. 读取用户材料，分开记录已确认产品事实、规范性要求和未知项。
3. 按第 4.1 节与用户确认产品需求；门禁未通过时提问并停止，不写文件。
4. 门禁通过后识别数据状态，明确评测目标、范围、成功标准和风险边界。
5. 从准确性、相关性、用户满意度或任务完成度等通用维度出发，结合产品真实任务、失败模式、受影响对象和风险，扩展为可以独立观察和评分的场景专属评测维度。通用维度只用于启发，不与专属维度并列形成第二套评分体系。
6. 从每个场景专属评测维度反推至少一类 Benchmark 用例，说明测试目的、输入形态、预期行为和典型失败模式。先建立“输入保真契约”，逐输入槽位记录真实用户操作、原始格式、Benchmark 表示、解析责任方和本轮覆盖层级，再生成或更新 `03_benchmark_cases.csv`。根据实际输入长度、结构、文件类型和复现需要，按需生成 `benchmark_materials/`；短且清晰的输入继续内联。
7. 使用与 `01_评测方案.md` 和 `03_benchmark_cases.csv` 完全一致的维度名称编写 Rubric，逐维度定义 3/2/1 分、具体示例、评分边界和否决关系。
8. 生成完整 `05_LLM-as-Judge评审方案.md`。
9. 在 `01_评测方案.md` 中记录已确认产品事实、已完成内容、下一步动作、待补数据和停止条件。
10. 资产生成后，若用户未说明是否执行，只询问一次是否“立即执行 / 仅准备执行包 / 暂不执行”；用户已明确只要资产时不重复询问。
11. 用户选择执行准备时，初始化 `run.json` 和按用例、运行序号预填充的 `results.csv`；用户暂不执行时不创建 `runs/`。
12. 用户选择立即执行时，Evaluation Runner 调用已确认的产品真实入口并记录真实候选结果。
13. 只有存在执行成功的真实候选输出后，才由当前调用本 Skill 的 Agent 执行 Judge。
14. 根据风险、异常、否决候选和抽样规则生成或更新 `human_review.csv`。
15. 按第 8 节检查报告门禁，生成初步分析或正式报告。
16. 将代表性失败案例写入 `07_bad_case_log.csv`，修复后复测并回流 Benchmark。
17. 维护 `08_持续评测机制.md`。只有命中 A/B 条件时才进入第 7 节专项分支。

“指导用户运行模型或回填数据”不代表已经产生评测结果。缺少下一阶段真实数据时，在当前门禁处停止并给出明确的下一步执行指引。

### 5.2 正式报告门禁

判定正式报告门禁是否满足时，必须同时检查：

- Benchmark、`run.json` 和成功的真实候选输出完整，且产品、模型或方案版本可追踪。
- Judge 分数、理由、证据和否决候选记录完整。
- 所有异常分、一票否决候选和实际生成的高风险代表复核任务均已完成人工复核。
- 普通样本已按预先定义的抽样规则完成人工复核。
- 人工复核结果和最终状态已回填 `human_review.csv`。
- 未解决的数据缺口、异常和限制已在报告中披露。

若用户未预先定义更严格规则，使用以下默认口径：

- 强制复核集合包括任一维度为 1 分、Judge 标记不确定、评分理由与证据矛盾、输出或评分字段缺失、一票否决候选，以及命中 Benchmark 或评测方案预先定义的可观察触发条件。P0/P1 和高风险标签本身不直接触发逐条复核。
- 未命中异常的 P0 或高风险 case，在同一 run 中稳定选择 1 条代表记录复核；多轮运行中同一 case 的同一维度最大分差达到 1 分、否决状态不一致，或不同 Judge 对同一维度的分差达到 1 分时，该 case 全部成功记录进入强制复核。
- 其余正常样本按“候选版本 × 用例类型”抽取 10%，每层不少于 2 条；该层不足 2 条时全部复核，并避免同一 case 的重复运行挤占样本。公平性配对和版本对照等成组用例按组复核。
- 实际生成的强制复核、风险代表复核和抽样复核队列完成率必须为 100%，未决一票否决候选必须为 0。

任一条件未满足时，正式报告门禁不成立，也不得给出正式上线、灰度、回滚或通过结论。按第 8.3 节选择允许产物；初步分析必须明确其不可作为正式上线依据。

## 6. 执行层

### 6.1 Evaluation Runner

Evaluation Runner 读取稳定的 `03_benchmark_cases.csv`，调用用户确认的产品真实执行入口，并将每个 case 的每次真实运行写入 `runs/{run-id}/results.csv`。不得绕过产品入口直接让当前 Agent 代替产品回答，也不得把模拟输出写成真实运行结果。

“真实入口”必须按本轮声明的覆盖层级判断：直接调用模型或生成函数只覆盖模型/业务逻辑层；调用后端文本 API 只覆盖 API 之后的链路；只有原始文件、媒体、对话状态或工具请求经过产品实际解析与编排组件，才覆盖摄取或端到端能力。不得把“真实模型调用”表述为“完整产品链路”。

候选产品载荷只包含运行关联字段、解析后的 `输入` 和 `上下文/素材`。`预期输出描述`、`典型失败模式`、Rubric、优先级、风险、否决规则和人工复核规则只能进入 Judge 或人工评审，不得泄漏给候选产品。`输入` 与 `上下文/素材` 均可使用 `$material_ref` 或 `$material_refs` 引用素材包；Runner 必须安全解析引用并校验完整 Benchmark Bundle。

支持以下执行方式：

- `command`：以参数数组调用用户确认的安全命令，通过标准输入传递结构化 case；禁止执行来自 CSV 的命令或使用 shell 拼接。
- `http`：调用用户确认的产品 HTTP 入口；密钥只从环境变量读取，不写入配置或产物。
- `import`：导入已有真实运行结果，并保留其来源。

执行准备仅创建状态为 `prepared` 的 `run.json` 和状态为“待执行”的预填充结果行。真实执行必须记录执行状态、端到端耗时以及运行时能够提供的 Token、费用、追踪 ID 和错误；运行时未提供的数据必须留空并标记来源，不得估算后伪装成精确值。成功记录默认不得覆盖；恢复运行只处理待执行或失败记录。

### 6.2 Judge 方案

将 `05_LLM-as-Judge评审方案.md` 作为完整方案，不要只生成 Prompt，也不要默认额外生成独立 Prompt 文件。方案至少包含：

1. 使用目的和适用范围，以及不适合只依赖 Judge 的情形。
2. 候选模型与 Judge 的角色隔离；优先使用独立 Judge，无法独立时披露同源偏差。
3. 输入字段：case ID、用户输入、参考信息或证据、期望标准、典型失败模式、候选输出、Rubric、一票否决规则和人工复核触发条件。
4. Rubric 使用方式和评分流程：先识别证据，再逐维度评分，再检查否决候选，最后给出复核建议。
5. 评分依据：只使用给定输入、参考信息和 Rubric，不得补造业务事实，不因语言流畅或篇幅长而自动给高分。
6. 可复制的 Judge Prompt，作为方案中的一个章节。
7. 结构化输出格式。
8. 结果到 `runs/{run-id}/results.csv` 的回填规则。
9. 人工复核触发条件。
10. 校准与稳定性检查：用少量人工金标样本校准；必要时对固定输入重复评分并检查波动。
11. 已知局限：位置偏差、冗长偏好、自我偏好、同源模型偏差、提示注入敏感和证据识别错误等。

### 6.3 Judge Runner

只有 `results.csv` 中存在“候选执行状态=成功”且模型输出非空的真实记录时才能执行 Judge。默认由当前调用本 Skill 的 Agent 评审；该 Agent 可以是 Codex、Claude Code 或其他具备所需能力的 Agent，不得将实现硬编码为某一宿主。Judge 输入应隔离候选生成过程，并尽量隐藏候选身份。逐条输出机器可解析结果，至少包含：

```json
{
  "case_id": "",
  "run_sequence": 1,
  "judge_model": "",
  "dimension_scores": {
    "dimension_name": {
      "score": 1,
      "reason": "",
      "evidence": ""
    }
  },
  "veto_candidate": {
```

将 Judge 执行状态、结构化分数、理由、不确定性、否决候选和复核原因回填 `results.csv`。Judge 阶段不得确认最终一票否决，只填写候选及理由；最终“是/否”仅写入人工复核文件。Judge 结果校验失败时应整体拒绝并返回错误，不得写入部分或不合法的 Judge JSON。Judge 执行只能支持初步分析，不能单独支持正式结论。

### 6.4 人工复核与最终裁决

按第 5.2 节建立强制复核集合和普通样本分层抽样集合；存在复核任务时才生成 `human_review.csv`。人工使用同一 Rubric，记录评分、理由、最终否决确认、复核人、复核状态、复核结果和最终状态。只有第 5.2 节门禁满足后，人工裁决才能汇总为正式报告和决策建议。

## 7. A/B 测试专项分支

A/B 测试不属于默认资产包，也不是离线评测主流程的必经步骤。

### 7.1 A/B 测试设计

命中第 8.3 节的 A/B 设计状态时，先收集必要信息。方案必须包含：

- 实验背景、改动点、决策目标、核心假设，以及对照组和实验组的唯一版本标识。
- 实验前离线准入：在同一固定 Benchmark 和运行条件下比较版本。
- 对随机性明显的生成任务进行多轮运行，比较均值、波动和高风险事件，而不是依赖单次结果。
- 在线随机分流：分流单位、随机化方式、用户范围、样本量或最小观察量、停止规则、周期和数据质量检查。
- 主指标、次要指标、护栏指标和一票否决事件。
- 离线与在线联合决策：离线质量决定能否进入实验，在线业务与用户指标决定是否值得扩量。
- 胜出、失败和不确定的判定标准，以及扩大灰度、继续观察、回滚和全量上线门槛。
- 风险事件、人工复核、紧急停止机制、数据记录要求和分析责任人。

此阶段的允许与禁止产物严格按第 8.3 节执行。

### 7.2 A/B 测试分析

命中第 8.3 节的 A/B 分析状态时，先校验组别、样本量、实验周期、随机化、主指标、护栏指标、风险事件和数据缺失，再给出条件化建议：

- 主指标改善、护栏正常、风险可接受且样本与周期充分：可建议扩大灰度或全量上线。
- 趋势有利但样本不足、周期不足、差异不稳定或关键字段缺失：输出“继续观察”或“暂不决策”。
- 主指标未改善且成本、质量或护栏恶化：建议停止扩量或回滚。
- 出现已确认的一票否决、高严重度错误、安全、合规或公平性风险：优先建议暂停扩量、人工复核或回滚。
- 没有预注册方案时可以做探索性分析，但必须披露证据限制，不得包装成确定性因果结论。

不要因为单个主指标提升而忽略护栏指标，也不要根据一次短周期实验宣布明确胜者。

## 8. 统一输出契约

本节是所有文件名和生成条件的唯一事实源。

### 8.1 保存与单文档规则

通过第 4.1 节产品事实确认门禁后，产物型请求默认保存到 `outputs/{product-slug}/`；若环境不可写，则在对话中给出完整内容并说明未实际保存。门禁通过前不得创建产品输出目录或写入评测文件。只有用户明确要求“不生成文件”“只讨论”或“先看草稿”时才不写文件。

用户只要求一个资产，包括只说“设计评测方案”时，只生成该文件，补充最小必要上下文、边界和依赖，不强制生成完整资产包。只有用户明确说“完整评测资产包”“全套评测资产”时才生成默认完整资产包。单一资产不足以支持决策时，明确说明仍缺少候选输出、评分或人工复核等步骤。

生成文件前必须读取该输出对应的模板，不得仅凭文件名或本 Skill 的摘要自行组织结构：

| 输出文件 | 强制模板 |
|---|---|
| `01_评测方案.md` | `templates/01_evaluation_plan_template.md` |
| `02_Benchmark用例设计.md` | `templates/02_benchmark_design_template.md` |
| `03_benchmark_cases.csv` | `templates/03_benchmark_cases_template.csv` |
| `04_Rubric评分标准.md` | `templates/04_rubric_template.md` |
| `05_LLM-as-Judge评审方案.md` | `templates/05_llm_as_judge_review_plan_template.md` |
| `06_评测报告模板.md` | `templates/06_evaluation_report_template.md` |
| `07_bad_case_log.csv` | `templates/07_bad_case_log_template.csv` |
| `07_BadCase分析报告模板.md` | `templates/07_bad_case_analysis_report_template.md` |
| `08_持续评测机制.md` | `templates/08_continuous_evaluation_template.md` |
| `benchmark_materials/manifest.json`（条件生成） | `templates/benchmark_material_manifest_template.json` |

模板一致性是产物门禁：

1. 除非用户明确要求“轻量版”，Markdown 输出必须保留对应模板的全部二级章节及其原始顺序；可以实例化内容，但不得合并、改名或删除必填章节。
2. 暂无信息的章节填写“待确认”“当前无真实数据”或“不适用”并说明原因，不得用删除章节掩盖信息缺口。
3. 模板中的必填表格、Judge JSON、Prompt、数据完整性和下一步指引必须实际展开，不能只写“参见统一结构”或用一句摘要代替。
4. CSV 表头必须与模板字段及顺序完全一致，不得自行增删列。
5. 生成或更新后运行 `scripts/validate_outputs.py --product-dir outputs/{product-slug}`；完整资产包追加 `--require-default-package`。校验失败时先修复，不得把未通过产物声明为完成。
6. 用户明确要求轻量版时，应在文件开头标记“轻量版”，并说明省略章节；轻量版不得冒充标准完整资产。

### 8.2 产品事实已确认且用户明确要求的默认完整资产包

```text
01_评测方案.md
02_Benchmark用例设计.md
03_benchmark_cases.csv
04_Rubric评分标准.md
05_LLM-as-Judge评审方案.md
07_bad_case_log.csv
08_持续评测机制.md
```

保留编号空位，不为追求连续编号而重命名现有资产。`07_bad_case_log.csv` 在此状态下只能是空台账，不得填入虚构案例。默认资产包不包含 `runs/`、报告模板、Bad Case 分析模板或任何 A/B 文件。

`benchmark_materials/` 是条件性 Benchmark 支撑资产，不是每个默认资产包的必选目录。只有输入较长、包含多文件、需要保留排版/代码/对话结构、使用二进制文件、需要复用素材或需要独立冻结素材时才生成；若生成，必须同时生成 `benchmark_materials/manifest.json`。

素材必须匹配本轮输入保真契约。内容完整度与载体保真度是两个独立要求：长 JSON 文本可以是完整的结构化 API 载荷，但不能冒充用户上传的完整 PDF；把 PDF 提前抽成文本只能评测解析后能力，不能评测上传、版面恢复、OCR 或文件解析。使用“合成素材”时，明确它是虚构而非压缩；只有具备该场景真实文档通常包含的章节、长度、噪声和结构时，才称为“完整素材”，否则称“片段”或“扩充文本”。

### 8.3 数据状态与允许产物

| 数据状态或请求 | 允许产物 | 禁止产物或结论 |
|---|---|---|
| 只有产品场景，产品事实未确认 | 对话中的产品事实摘要和澄清问题 | 创建输出目录、任何评测文件、评分或决策结论 |
| 产品事实已确认，无真实评测数据 | 用户点名的单项资产；明确要求全套时生成默认资产包 | 正式分数、初步/正式报告、上线决策 |
| 用户选择执行准备 | `runs/{run-id}/run.json` 和预填充的 `results.csv` | 模型输出、评分、分析或决策结论 |
| 评测运行执行中或部分失败 | 更新对应 run 的 `run.json` 和 `results.csv` | 把待执行或失败记录当作真实结果；初步/正式报告 |
| 有成功候选输出，无 Judge 结果 | 更新对应 run 的 `results.csv` | 初步/正式报告和决策结论 |
| 有 Judge 结果，人工复核不足 | `06_初步评测分析.md`；更新 `results.csv`；有复核任务时生成或更新 `human_review.csv` | `06_评测报告.md` 和正式上线依据 |
| 第 5.2 节门禁全部满足 | `06_评测报告.md` | 用平均分覆盖已确认否决项 |
| 用户明确要求报告模板 | `06_评测报告模板.md` | 真实分数、真实结论或伪装成已执行评测 |
| 有真实 Bad Case | 更新 `07_bad_case_log.csv`；按触发条件生成 `07_BadCase阶段分析报告.md` | 虚构案例；普通案例默认一例一文档 |
| 明确要求 A/B、无真实实验数据 | `AB测试方案.md` | A/B 分析、扩量、回滚或全量结论 |
| 有真实 A/B 数据 | `AB测试分析报告.md` | 数据不足时给确定性胜负结论 |

### 8.4 Benchmark 与运行数据

`03_benchmark_cases.csv` 只保存稳定的用例和规则字段：用例编号、类型、优先级、评测维度、输入、上下文或素材、预期输出描述、典型失败模式、风险等级、一票否决规则、人工复核触发条件和备注。不要把候选输出、Judge 或人工字段重新追加到该文件。

`输入` 表示用户直接要求产品完成的任务；`上下文/素材` 表示完成任务所需的背景、文档、数据或附件。两个字段均支持：

- 直接内联短文本或结构化 JSON。
- `{"$material_ref":"benchmark_materials/CASE-001/input.md"}` 单文件引用。
- `{"$material_refs":{"document":"benchmark_materials/CASE-001/document.md"}}` 多个命名素材引用。

Runner 对文本素材展开为内容；对 PDF、图片、音频、视频等二进制素材传递含安全绝对路径、媒体类型和 SHA-256 的文件描述，由产品适配器通过真实产品入口处理，不得由 Runner 预解析后冒充产品能力。素材路径必须相对 Benchmark 目录且不得越界。`manifest.json` 必须列出所有且仅列出被引用文件及摘要；CSV、Manifest 和全部素材共同构成冻结的 Benchmark Bundle。

生成素材前执行以下保真门禁：

1. 为每个输入槽位选择覆盖层级：模型/业务逻辑层、解析后 API 层、摄取/编排层或用户端到端层。
2. 若本轮要评测文件解析、媒体理解、表格公式、代码仓库、工具调用或多轮状态，素材必须保留对应原始载体和结构，不得用抽取文本或汇总 JSON 代替。
3. 若仅评测解析后能力，可以使用 Markdown、文本或结构化 JSON，但必须在 `01_评测方案.md` 和 `02_Benchmark用例设计.md` 明示未覆盖的上游链路，交付时不得称为端到端评测。
4. 在执行前验证适配器能消费每种素材描述并调用声明层级的真实组件。若不能，停止并报告“入口能力缺口”；不得为了跑通而静默改格式、预解析二进制文件或绕过产品组件。
5. 对照一名真实用户手动完成同一任务时会提供的全部输入，检查字段、文件数量、格式、长度、版式、噪声和状态是否具有代表性。任何有意简化都必须记录原因及其对可信度的影响。

只有用户选择“立即执行”或“仅准备执行包”时才创建 `runs/{run-id}/`：

```text
runs/{run-id}/
├── run.json
├── results.csv
├── human_review.csv      # 有复核任务时生成
└── raw_results.jsonl     # 复杂响应或审计需要时生成
```

- `run.json` 保存本轮共享元数据、版本、执行方式、时间、状态和配置指纹，不保存密钥。
- `results.csv` 使用“一条 case × 一次运行一行”的长表，保存候选执行和 Judge 结果。
- `human_review.csv` 保存人工评分、最终否决确认和最终结论，不与 Judge 候选字段混用。
- `raw_results.jsonl` 仅在产品响应无法安全压入 CSV 或用户要求原始审计时生成。

默认每轮只创建 `run.json` 和 `results.csv`。通过“评测运行 ID + 用例编号 + 运行序号”关联运行文件，通过用例编号关联 Benchmark。比较多个产品版本时，每个版本使用独立 run，避免在同一结果表中混杂运行级配置。

`runner_config.json` 不属于默认评测资产。只有用户选择执行且需要复用产品入口配置时才保存；不得包含明文密钥，run 中引用的 Benchmark、Rubric 和其他本地文件路径必须能从 run 目录正确解析。

保持 `07_bad_case_log.csv` 为长期维护的精简台账。仅将 P0/P1、高风险、反复出现、线上真实反馈、影响核心任务或能代表典型失败模式的案例沉淀入台账。阶段分析基于真实台账生成。默认不要为每条 Bad Case 生成独立 Markdown；仅 P0、重大线上问题、合规风险或用户明确要求时例外。

## 9. 产品类型与评测重点

- RAG / 知识库问答：事实准确性、检索相关性、引用充分性与可追溯性、上下文使用、不确定性、过期知识和幻觉风险。
- Agent / 多工具调用：任务规划、工具选择与参数、中间状态、异常恢复、任务完成度、权限安全、误操作和高风险确认。
- AI 写作 / 内容生成：意图符合度、结构、风格、可读性、事实与引用、原创性、品牌调性和空泛套话。
- AI 简历筛选 / 审核 / 推荐：依据充分性、一致性、公平性、可解释性、合规、偏见、错杀漏判和敏感因素使用。
- AI 客服 / 对话助手：问题理解、准确性、流程引导、语气、人工升级、投诉误导和不可兑现承诺。
- AI 数据分析 / 报告生成：数据理解、计算、结论、图表解释、异常值、不确定性和相关性与因果性的区分。
- AI 编程助手：需求理解、正确性、可运行性、安全、可维护性、测试、隐藏副作用和未授权修改。
- 多模态 AI 产品：跨模态理解、证据使用、媒体一致性、识别置信度、敏感内容处理和可解释性。

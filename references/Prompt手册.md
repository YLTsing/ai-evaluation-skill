# 常用场景与 Prompt 手册

本文档提供调用 `ai-evaluation-skill` 的常用场景和可复制 Prompt。它是用户操作参考，不替代 `SKILL.md`、模板或工作流中的数据门禁。

## 使用约定

- 将 `{花括号内容}` 替换为真实信息；不确定的信息保留为“待确认”，不要让 Agent 自行补造产品事实。
- “生成资产”不等于“执行评测”。只有明确要求运行，并确认真实产品入口后，Evaluation Runner 才能调用产品。
- 产品入口可以连接任意模型 API、本地模型、工作流、工具或混合架构，不绑定特定模型厂商。
- Judge 默认由当前调用本 Skill 的 Agent 执行，例如 Codex、Claude Code 或其他 Agent；它只能标记一票否决候选，不能替代人工最终裁决。
- 正常执行请求默认先完成执行前检查并展示预计调用范围；在第一次真实调用产品前等待确认。只有用户明确写出“预检通过后无需二次确认”时，才可连续执行。
- 没有成功的真实候选输出，不得执行 Judge；人工复核门禁未满足时，不得生成正式报告或上线结论。
- 默认文件保存在 `outputs/{product-slug}/`。运行数据保存在 `outputs/{product-slug}/runs/{run-id}/`。

## Prompt 编写建议

一个稳定的请求通常应包含：产品路径或名称、目标模式、事实来源、评测范围、目标版本、是否允许写文件、是否允许真实调用、运行次数和期望停止点。涉及执行时，还应说明产品入口和密钥环境变量是否已准备好。

## 一、评测资产设计

### 场景 1：首次梳理产品与评测需求

**适用时机**：只有产品介绍、代码、PRD、知识库或 Prompt，尚未确认真实输入输出和工作流。

**最低输入**：产品名称或路径，以及至少一种产品材料。

**可复制 Prompt**：

```text
请使用 ai-evaluation-skill 梳理 {产品名称或路径} 的评测需求。
先阅读我提供的 {代码/PRD/知识库/Prompt/产品说明}，区分“已确认产品事实”“材料中的规范要求”和“尚未确认项”。
请先输出产品事实摘要，并只追问会影响指标、Benchmark、Rubric、风险规则或执行入口的必要问题。
在我确认前不要创建 outputs 目录，也不要生成评测文件。
```

**预期产物**：对话中的产品事实摘要与 5–8 个必要问题。

**不会执行的动作**：不生成资产，不创建 run，不调用产品，不执行 Judge。

**关键数据门禁**：产品事实摘要必须经用户确认，或用户明确允许按假设生成草案。

### 场景 2：生成完整评测资产包（高频）

**适用时机**：产品事实已确认，需要一套可直接进入执行准备的标准资产。

**最低输入**：确认后的用户、任务、输入、输出、工作流、风险、评测目标和范围。

**可复制 Prompt**：

```text
请为 {产品名称} 生成完整评测资产包，产品目录使用 {product-slug}。
以下是已确认产品事实：{粘贴事实摘要}。
评测目标是 {上线前准入/版本对比/持续评测}，目标版本是 {版本}。
严格读取并遵循 templates 中对应模板，生成默认资产包并运行输出校验。
本次只生成资产，不创建 runs，不调用产品，不执行 Judge，也不生成报告或结论。
```

**预期产物**：`01_评测方案.md`、`02_Benchmark用例设计.md`、`03_benchmark_cases.csv`、`04_Rubric评分标准.md`、`05_LLM-as-Judge评审方案.md`、`07_bad_case_log.csv`、`08_持续评测机制.md`。

**不会执行的动作**：不生成报告模板、A/B 文件或运行附件；不调用真实产品。

**关键数据门禁**：只有明确要求“完整”或“全套”时才生成完整资产包。

### 场景 3：只生成一项评测资产

**适用时机**：只需要评测方案、Benchmark、Rubric、Judge 方案或持续评测机制中的一项。

**最低输入**：已确认产品事实，以及该资产依赖的目标、范围或指标。

**可复制 Prompt**：

```text
请只为 {产品名称} 生成 {01_评测方案.md/03_benchmark_cases.csv/04_Rubric评分标准.md/05_LLM-as-Judge评审方案.md/08_持续评测机制.md}。
已确认产品事实：{事实摘要}。
请读取对应模板，保持标准字段、二级章节和顺序，并保存到 outputs/{product-slug}/。
不要自动补齐完整资产包；在文件中轻量说明依赖、边界和下一步。
```

**预期产物**：仅用户点名的文件。

**不会执行的动作**：不生成未点名文件，不创建 run，不产生评测结论。

**关键数据门禁**：若信息不足，先确认事实；用户允许草案时必须显著标记假设。

### 场景 4：审查现有评测资产

**适用时机**：已有一份或多份资产，需要检查可执行性、门禁和模板一致性。

**最低输入**：待审查文件或产品输出目录。

**可复制 Prompt**：

```text
请审查 {文件或目录路径}，不要先修改。
重点检查：是否符合 templates、字段和文件名是否一致、产品事实是否被假设替代、Benchmark 是否可执行、Rubric 是否可判分、Judge 输入输出是否闭合、报告和人工复核门禁是否存在绕过路径。
请按严重度列出问题、对应文件位置和可执行修改建议。
```

**预期产物**：对话中的审查结果；用户明确要求落盘时再生成审查文档。

**不会执行的动作**：不默认修改文件，不运行产品，不生成新结果。

**关键数据门禁**：审查已有内容属于非产物型请求，默认不写文件。

### 场景 5：增量更新 Benchmark

**适用时机**：增加新业务、边界、风险、回归或真实 Bad Case 用例。

**最低输入**：现有 Benchmark、变更事实或真实失败案例。

**可复制 Prompt**：

```text
请增量更新 outputs/{product-slug}/03_benchmark_cases.csv。
新增来源是 {需求变更/风险规则/真实 Bad Case/线上反馈}：{内容}。
保留现有用例编号和 12 列 Schema，不覆盖历史用例；为新增用例分配稳定编号，并检查与 Rubric、否决规则和人工复核条件的对应关系。
不要在 Benchmark 中追加候选输出、Judge 或人工字段。
```

**预期产物**：更新后的 `03_benchmark_cases.csv`，必要时同步更新 `02_Benchmark用例设计.md` 或 Rubric，但仅在用户授权后执行。

**不会执行的动作**：不创建 run，不把虚构案例写成真实 Bad Case。

**关键数据门禁**：真实 Bad Case 必须有真实来源；仅设计的新风险用例应标记为设计用例。

## 二、候选执行

### 场景 6：只准备评测运行包

**适用时机**：Benchmark 已冻结，希望先检查运行规模和附件，但暂不调用产品。

**最低输入**：产品目录、产品版本、运行次数和执行方式。

**可复制 Prompt**：

```text
请为 outputs/{product-slug}/03_benchmark_cases.csv 准备一轮评测运行包。
产品版本为 {版本}，每条用例运行 {N} 次，执行方式为 {command/http/import}，run-id 使用 {run-id 或自动生成}。
只执行 Evaluation Runner 的 init：生成 run.json 和预填充 results.csv。
请汇总预计用例数、总运行行数和缺失配置；不要调用产品，不要执行 Judge。
```

**预期产物**：`runs/{run-id}/run.json` 和预填充的 `results.csv`。

**不会执行的动作**：不产生模型输出、Token、费用、Judge 分数或报告。

**关键数据门禁**：Benchmark Schema 有效，产品版本和运行次数明确。

### 场景 7：启动真实产品评测（高频）

**适用时机**：需要通过产品真实入口批量跑 Benchmark，同时希望在真实调用前确认成本与范围。

**最低输入**：已冻结 Benchmark、产品版本、真实产品入口、运行次数和必要环境变量。

**可复制 Prompt**：

```text
请为 {产品路径或产品名称} 启动真实评测，使用 outputs/{product-slug}/03_benchmark_cases.csv。
产品版本是 {版本}，每条用例运行 {N} 次，入口方式是 {command/http}：{入口或配置说明}。
先完成执行前检查，向我展示：run-id、用例数、总调用数、入口与配置摘要、超时、重试、可获得的 Token/费用字段、密钥环境变量状态和预计停止点。
在第一次真实调用产品前暂停，等我确认后再运行；不要绕过产品入口让当前 Agent 代答。
候选执行完成后先汇总成功、失败和缺失字段，不要自动执行 Judge。
```

**预期产物**：确认前不产生真实调用；确认后生成或更新 `run.json` 与 `results.csv`。

**不会执行的动作**：不让当前 Agent 代替产品生成候选输出，不自动进入 Judge 或报告阶段。

**关键数据门禁**：产品入口连通、配置无明文密钥、用户完成真实调用确认。

### 场景 8：参数明确后连续执行，不做二次确认

**适用时机**：用户已经核对调用范围、成本和入口，并明确授权本轮直接执行。

**最低输入**：场景 7 的全部信息，以及明确的连续执行授权。

**可复制 Prompt**：

```text
请立即执行 {product-slug} 的真实候选评测。
我已确认：产品版本 {版本}、Benchmark {路径}、每条运行 {N} 次、入口配置 {路径或说明}、超时 {秒}、重试 {次数}，并已准备好所需环境变量。
预检通过后无需二次确认，直接调用产品入口；若入口检查失败、配置指纹变化、预计调用数与 {数量} 不一致或发现明文密钥，立即停止。
执行结束后只汇总 run 状态，不自动执行 Judge。
```

**预期产物**：本轮 `run.json` 与真实 `results.csv`。

**不会执行的动作**：门禁异常时不强行继续，不自动 Judge。

**关键数据门禁**：用户明确授权“预检通过后无需二次确认”，且预检结果与授权范围一致。

### 场景 9：恢复失败或中断的运行

**适用时机**：run 为 `partial`、`failed` 或存在待执行记录。

**最低输入**：`runs/{run-id}/` 和原执行配置。

**可复制 Prompt**：

```text
请检查 outputs/{product-slug}/runs/{run-id}/ 的失败原因和可恢复性。
先列出成功、失败、待执行数量，并核对 runner_config.json 的配置指纹和产品入口。
确认后使用 resume 只重跑失败或待执行记录，保留所有成功结果；不要创建新的候选答案覆盖成功行。
恢复完成后汇总状态和仍未解决的错误，不执行 Judge。
```

**预期产物**：更新后的 `run.json` 和 `results.csv`。

**不会执行的动作**：不覆盖成功记录，不隐去历史错误，不自动 Judge。

**关键数据门禁**：入口和配置仍对应同一产品版本；若版本改变，应创建新 run。

### 场景 10：导入已有真实候选输出

**适用时机**：候选输出由外部系统完成，需要进入统一运行数据结构。

**最低输入**：真实结果文件、产品版本、来源说明和字段映射。

**可复制 Prompt**：

```text
请将 {真实结果文件路径} 导入 {product-slug} 的评测运行数据。
这些结果来自 {真实系统和运行时间}，产品版本为 {版本}，字段映射为 {用例编号/运行序号/模型输出/耗时/Token/费用/追踪ID}。
使用 import-results，不把缺失字段估算为精确值；保留来源和错误信息。
导入后检查用例覆盖率和重复记录，不执行 Judge。
```

**预期产物**：一个可追踪的 run，以及导入后的 `results.csv`。

**不会执行的动作**：不调用产品，不把示例或人工编造文本当成真实输出。

**关键数据门禁**：来源可追踪，至少能关联用例编号、运行序号和真实模型输出。

## 三、Judge 与人工复核

### 场景 11：由当前 Agent 执行 Judge（高频）

**适用时机**：run 中已有成功、非空的真实候选输出。

**最低输入**：`results.csv`、Rubric 和完整 Judge 方案。

**可复制 Prompt**：

```text
请对 outputs/{product-slug}/runs/{run-id}/ 中执行成功且输出非空的记录进行 Judge。
严格使用 outputs/{product-slug}/04_Rubric评分标准.md 和 05_LLM-as-Judge评审方案.md。
由当前调用 Skill 的 Agent 逐条评分：先 prepare 隔离输入，再输出符合 Schema 的结构化 JSON，校验通过后 apply 回填 results.csv。
Judge 只能标记“一票否决候选”和人工复核原因，不得确认最终否决或填写人工结论。
结束后汇总 Judge 成功、失败、否决候选和待人工复核数量。
```

**预期产物**：更新后的 `results.csv`；存在复核任务时生成 `human_review.csv`。

**不会执行的动作**：不评价失败或空输出，不越权给最终否决和正式上线结论。

**关键数据门禁**：成功真实输出、Rubric 和 Judge 方案齐全；Judge JSON 必须整体校验通过。

### 场景 12：生成并完成人工复核任务

**适用时机**：已有 Judge 结果，需要处理强制复核和普通样本抽样。

**最低输入**：Judge 完成的 run、复核规则和人工评审人信息。

**可复制 Prompt**：

```text
请检查 outputs/{product-slug}/runs/{run-id}/ 的人工复核门禁。
按 SKILL.md 的强制复核规则和“候选版本 × 用例类型”分层抽样规则，生成或更新 human_review.csv。
先列出强制集合、抽样集合、重复项和完成率，不替人工填写分数、最终否决或结论。
我填回人工结果后，再检查强制复核完成率、抽样复核完成率和未决否决候选是否均满足正式报告门禁。
```

**预期产物**：`human_review.csv` 及门禁状态摘要。

**不会执行的动作**：不虚构人工身份、评分、复核时间或最终裁决。

**关键数据门禁**：强制复核与抽样复核均 100% 完成，未决否决候选为 0，才可进入正式报告。

### 场景 13：查看运行与报告门禁状态

**适用时机**：想知道本轮停在哪一步、下一步做什么。

**最低输入**：`runs/{run-id}/`。

**可复制 Prompt**：

```text
请检查 outputs/{product-slug}/runs/{run-id}/ 的当前状态。
运行 Judge Runner 的 status，并汇总候选执行、Judge、强制人工复核、抽样人工复核和未决否决候选状态。
明确当前允许生成什么、禁止生成什么，以及解除门禁所需的最小下一步。
不要修改运行数据。
```

**预期产物**：对话中的状态和下一步清单。

**不会执行的动作**：不调用产品，不执行 Judge，不生成报告。

**关键数据门禁**：以实际 run 文件为准，不以对话中的口头描述代替数据检查。

## 四、分析、报告与比较

### 场景 14：生成初步评测分析（高频）

**适用时机**：已有 Judge 结果，但人工复核门禁尚未全部满足。

**最低输入**：真实候选输出和 Judge 结果。

**可复制 Prompt**：

```text
请基于 outputs/{product-slug}/runs/{run-id}/ 生成 06_初步评测分析.md。
先核对候选与 Judge 数据完整性，披露人工复核缺口、未决否决候选和限制。
只总结初步分布、风险信号和待复核项，不给正式通过、上线、灰度、回滚或最终否决结论。
请显著标注“基于 Judge 的初步结果，不可作为正式上线依据”。
```

**预期产物**：`06_初步评测分析.md`。

**不会执行的动作**：不生成 `06_评测报告.md`，不把平均分当作上线结论。

**关键数据门禁**：至少有真实候选和有效 Judge 结果；无 Judge 时连初步分析也不生成。

### 场景 15：生成正式评测报告（高频）

**适用时机**：候选、Judge、强制复核与抽样复核均已完成。

**最低输入**：完整 run、有效 Judge 结果和完成的 `human_review.csv`。

**可复制 Prompt**：

```text
请检查 outputs/{product-slug}/runs/{run-id}/ 是否满足正式报告门禁。
只有强制复核完成率=100%、抽样复核完成率=100%、未决否决候选=0，且版本与数据来源可追踪时，才生成 06_评测报告.md。
报告必须区分 Judge 与人工结果，单列已确认否决项、高优先级用例、数据缺口和决策理由；平均分不得覆盖事实、安全、公平或合规风险。
若门禁不满足，请停止并列出缺口，不要降格伪装成正式报告。
```

**预期产物**：门禁满足时生成 `06_评测报告.md`；否则只输出缺口说明。

**不会执行的动作**：门禁不足时不生成正式报告或决策结论。

**关键数据门禁**：严格使用 `SKILL.md` 第 5.2 节，不以“已有部分人工结果”替代完成门禁。

### 场景 16：比较多个产品版本

**适用时机**：比较模型、Prompt、RAG、Agent 或工作流版本。

**最低输入**：同一冻结 Benchmark、同一 Rubric、每个版本独立 run 和可比运行条件。

**可复制 Prompt**：

```text
请比较 {run-id-A} 与 {run-id-B}，对应版本为 {版本A} 和 {版本B}。
先检查 Benchmark、Rubric、运行次数、输入上下文和执行条件是否可比；不可比项必须披露。
比较逐维度结果、高优先级用例、波动、否决候选或最终否决、延迟、Token、费用和稳定性。
不要因一次平均分差异直接宣布胜者；正式版本建议必须满足各自人工复核门禁。
```

**预期产物**：用户明确要求时生成版本对比分析或报告，否则在对话中给出结果。

**不会执行的动作**：不把不同 Benchmark 或配置下的分数作直接因果比较。

**关键数据门禁**：输入和评审条件可比；正式建议需满足人工复核门禁。

## 五、Bad Case 与持续评测

### 场景 17：沉淀真实 Bad Case（高频）

**适用时机**：发现值得长期跟踪的真实失败案例。

**最低输入**：真实来源、输入、上下文、实际输出、期望行为、版本和严重度依据。

**可复制 Prompt**：

```text
请将以下真实失败案例评估后写入 outputs/{product-slug}/07_bad_case_log.csv：{案例信息}。
先判断它是否属于 P0/P1、高风险、反复出现、真实用户反馈、影响核心任务或典型失败模式。
保留精简台账 Schema，填写来源、关联 run/case、失败类型、根因层、修复建议、状态和是否加入 Benchmark。
不要默认为单条案例生成 Markdown；不要补造缺失事实。
```

**预期产物**：更新后的 `07_bad_case_log.csv`；必要时提出回归用例建议。

**不会执行的动作**：不把所有低价值失败都纳入台账，不虚构 Bad Case。

**关键数据门禁**：案例必须来自真实评测、线上反馈或可追踪记录。

### 场景 18：生成阶段性 Bad Case 分析

**适用时机**：按评测轮次、月份、版本或风险专题复盘真实台账。

**最低输入**：有真实记录的 Bad Case 台账和明确分析范围。

**可复制 Prompt**：

```text
请基于 outputs/{product-slug}/07_bad_case_log.csv 生成 07_BadCase阶段分析报告.md。
分析范围为 {时间/版本/run-id/模块/风险专题}。
按失败类型、严重度、风险等级、根因层和状态汇总，提炼高频模式、修复优先级、应回流 Benchmark 的案例和回归建议。
不要把台账空白或未确认项推断成事实，也不要默认一例一文档。
```

**预期产物**：`07_BadCase阶段分析报告.md`。

**不会执行的动作**：无真实台账时不生成阶段结论；普通案例不单独生成复盘文件。

**关键数据门禁**：报告范围内必须存在足够的真实 Bad Case 记录。

### 场景 19：建立或更新持续评测机制

**适用时机**：需要定义变更触发、定期回归、告警和责任人。

**最低输入**：产品变更类型、发布节奏、风险等级、快速与完整 Benchmark 范围。

**可复制 Prompt**：

```text
请为 {产品名称} 生成或更新 outputs/{product-slug}/08_持续评测机制.md。
已确认的变更来源包括 {模型/Prompt/检索/工具/预处理/策略/交互}，发布节奏为 {节奏}，风险等级为 {等级}。
请定义变更触发和定期触发、快速回归与完整 Benchmark、Bad Case 回流、Rubric 版本、告警阈值、角色分工和决策记录。
只设计机制，不虚构历史运行结果或告警事件。
```

**预期产物**：`08_持续评测机制.md`。

**不会执行的动作**：不自动创建定时任务，不运行产品。

**关键数据门禁**：机制中的阈值应有业务依据；没有依据时标记待确认。

## 六、A/B 测试专项分支

### 场景 20：设计 A/B 测试（高频）

**适用时机**：明确提出线上 A/B 测试，但尚无真实实验数据。

**最低输入**：改动、对照组、实验组、假设、分流单位、指标、周期和风险。

**可复制 Prompt**：

```text
请为 {产品名称} 设计 A/B 测试，目前没有真实实验数据。
改动是 {改动}，对照组 {A}，实验组 {B}，核心假设 {假设}，分流单位 {单位}，用户范围 {范围}。
请先补齐必要信息，再只生成 outputs/{product-slug}/AB测试方案.md。
方案必须包含固定 Benchmark 输入、随机任务多轮运行及均值与波动、实验前离线准入、线上随机分流、主指标、护栏指标、样本与周期、扩量/观察/回滚/全量门槛和离线在线联合决策。
不要生成分析报告或任何实验结论。
```

**预期产物**：`AB测试方案.md`。

**不会执行的动作**：不生成 `AB测试分析报告.md`，不建议实际扩量或回滚。

**关键数据门禁**：没有真实数据时严格停留在设计分支。

### 场景 21：分析真实 A/B 测试数据（高频）

**适用时机**：已有真实实验数据，需要条件化决策建议。

**最低输入**：实验方案、组别、样本量、周期、随机化、主指标、护栏指标和风险事件。

**可复制 Prompt**：

```text
请基于 {实验方案路径} 和 {真实 A/B 数据路径} 生成 outputs/{product-slug}/AB测试分析报告.md。
先检查组别、样本量、周期、随机化、数据缺失、主指标、护栏指标和风险事件是否足以决策。
只有证据充分时才给扩大灰度或全量建议；样本不足、周期不足、差异不稳定或关键字段缺失时输出“继续观察”或“暂不决策”。
出现已确认否决、安全、合规、公平性或高严重度风险时，优先建议暂停扩量、人工复核或回滚。
不要因单个主指标提升忽略护栏恶化。
```

**预期产物**：`AB测试分析报告.md`。

**不会执行的动作**：数据不足时不宣布胜者，不把探索性分析包装成因果结论。

**关键数据门禁**：真实实验数据充分、可追踪，且决策标准与风险规则明确。

## 七、维护、校验与排障

### 场景 22：校验输出与模板一致性

**适用时机**：生成或更新资产后，检查章节、CSV Schema、run 路径和配置安全。

**最低输入**：产品输出目录。

**可复制 Prompt**：

```text
请校验 outputs/{product-slug}/ 的产物一致性。
运行 scripts/validate_outputs.py --product-dir outputs/{product-slug}；若这是完整资产包，追加 --require-default-package。
列出所有错误及对应文件，修复时保持模板二级章节顺序、CSV 字段顺序和现有真实数据。
修复后重新运行校验，并报告最终结果。
```

**预期产物**：修复后的资产和校验结果；若用户只要求检查，则不修改。

**不会执行的动作**：不为通过校验而删除真实数据或降低门禁。

**关键数据门禁**：完整资产包必须通过 `--require-default-package`。

### 场景 23：迁移旧版宽表 Benchmark

**适用时机**：旧 `03_benchmark_cases.csv` 混有候选、Judge 或人工字段。

**最低输入**：旧宽表路径和新的输出目录。

**可复制 Prompt**：

```text
请非破坏性迁移 {旧版CSV路径} 到 {新输出目录}。
使用 evaluation_runner.py migrate，将稳定用例字段保留在新的 03_benchmark_cases.csv，将历史候选、Judge 和人工数据迁入 legacy-import runs。
不要覆盖源文件；迁移后检查版本分组、字段映射、空值和关联键，并列出需要人工确认的项目。
```

**预期产物**：新的精简 Benchmark 和 `legacy-import` 运行目录。

**不会执行的动作**：不原地改写旧表，不假定旧字段一定可信。

**关键数据门禁**：迁移后必须人工检查版本和字段映射，历史数据不自动获得正式报告资格。

### 场景 24：诊断 Evaluation Runner 失败

**适用时机**：真实运行全部或部分失败，需要定位入口、配置、TLS、超时或响应映射问题。

**最低输入**：失败 run、执行配置、产品入口日志或错误信息。

**可复制 Prompt**：

```text
请诊断 outputs/{product-slug}/runs/{run-id}/ 的 Evaluation Runner 失败。
先只读检查 run.json、results.csv、runner_config.json 和可用产品日志，按配置、入口连通、认证、TLS、超时、重试、请求格式、响应 JSON、response_mapping 和产品内部错误分类。
请给出证据、影响范围和最小修复建议；不要把失败行当成真实结果，不要自动重跑。
需要真实联网调用、修改配置或恢复运行前，先说明拟执行动作和影响并等待确认。
```

**预期产物**：故障分类、证据和修复建议；用户确认后才更新配置或恢复 run。

**不会执行的动作**：不隐式关闭 TLS 校验，不泄露密钥，不自动重跑造成额外费用。

**关键数据门禁**：修复后必须重新做入口预检；若产品版本或行为发生变化，应创建新 run 而非混写旧结果。

## 高频入口速查

最常用的八个入口是：

1. 场景 2：生成完整评测资产包。
2. 场景 7：预检并启动真实产品评测。
3. 场景 11：由当前 Agent 执行 Judge。
4. 场景 14：人工门禁不足时生成初步分析。
5. 场景 15：门禁满足后生成正式报告。
6. 场景 17：沉淀真实 Bad Case。
7. 场景 20：只设计 A/B 测试。
8. 场景 21：分析真实 A/B 数据。

推荐把一次完整评测拆成独立确认点：产品事实确认 → 资产生成 → 执行预检与授权 → 候选执行 → Judge → 人工复核 → 报告。任一阶段缺少真实输入时，应停在当前门禁，不用空文件或模拟数据伪装流程已经完成。

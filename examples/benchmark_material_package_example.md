# Benchmark 用例素材包示例

只有输入较长、包含多文件、需保留结构、使用二进制附件或需要独立冻结时才生成素材包。短输入继续写在 CSV 中。

## 通用结构

```text
outputs/product/
├── 03_benchmark_cases.csv
└── benchmark_materials/
    ├── manifest.json
    └── CASE-001/
        ├── input.md
        └── document.pdf
```

单文件引用：

```json
{"$material_ref":"benchmark_materials/CASE-001/input.md"}
```

多个命名素材：

```json
{"$material_refs":{"policy":"benchmark_materials/CASE-001/policy.md","attachment":"benchmark_materials/CASE-001/document.pdf"}}
```

## 先选择覆盖层级

- 模型/业务逻辑层：可以传解析后的文本或结构化 JSON，只评价下游生成与判断。
- 解析后 API 层：使用 API 实际接收的字段和类型，不评价上游文件或媒体解析。
- 摄取/编排层：保留 PDF、图片、表格、代码仓库、工具状态等原始载体，由产品实际组件解析。
- 用户端到端层：还要经过真实 UI、上传、多轮状态或交互流程。

适配器只能接受文本时，不得把 PDF 预解析后声称覆盖了 PDF 功能；应停止执行并报告入口能力缺口。

## 场景映射

- 招聘：`输入` 为 HR 任务，`上下文/素材` 分别引用 JD 与简历。若真实路径为填写 JD、上传 PDF 简历，摄取层或端到端用例应使用 `{"$material_refs":{"jd":".../jd.md","resume":".../resume.pdf"}}`；仅测解析后判断时才使用文本，并明确不覆盖 PDF。
- RAG：`输入` 为用户问题，`上下文/素材` 引用知识文档。
- 客服：`输入` 为当前问题，`上下文/素材` 引用历史对话和政策。
- 编程：`输入` 为修改需求，`上下文/素材` 引用代码、日志和测试。
- 数据分析：`输入` 为分析问题，`上下文/素材` 引用数据集。
- Agent：`输入` 为任务，`上下文/素材` 引用初始状态和工具描述。
- 多模态：`输入` 为文本指令，`上下文/素材` 引用图片、音频或视频。

候选产品载荷只包含解析后的 `input` 和 `context` 及运行关联字段。预期输出、典型失败模式、Rubric、优先级、风险和否决规则只进入 Judge 或人工评审。

“合成”表示素材由评测设计者虚构，不表示压缩。只有结构、章节、长度、版式和噪声达到该场景常见真实文档水平时，才称“完整合成文档”。

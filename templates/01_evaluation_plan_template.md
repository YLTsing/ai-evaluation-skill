## 核心评测指标

评测指标用于在打分之前定义“什么是好”。可以从以下通用维度出发，但不能直接照搬；应结合产品任务继续拆成下一节的场景专属评测维度。

### 准确性

- 核心问题：输出中的事实、逻辑、计算或操作是否正确？
- 在本产品中的具体含义：`结合产品场景填写`

### 相关性

- 核心问题：输出是否真正回应用户任务并使用了正确上下文？
- 在本产品中的具体含义：`结合产品场景填写`

### 用户满意度 / 任务完成度

- 核心问题：用户能否基于输出完成想做的工作？
- 在本产品中的具体含义：`结合产品场景填写`

设计时还应检查：同一个通用维度是否包含不同失败模式，除了直接用户还有谁会被输出影响，以及哪些安全、公平、合规、隐私或业务风险需要单独评测。

## 场景专属评测维度

根据产品的真实任务和失败模式，将通用维度扩展成可以独立观察和评分的专属评测维度。不要把“准确性高”“体验好”这类抽象口号直接作为最终维度。

每个专属维度都要回答四件事：它具体测什么、评审时要问什么、为什么需要从通用维度中单独拆出，以及它可能暴露什么失败。

- 无一票否决项触发，或触发项已修复并复测通过。

## 下一步执行指引

1. 检查 Benchmark 用例是否覆盖核心场景、边界场景、风险场景和已知易失败场景。
2. 若用户未说明是否执行，资产生成后只询问一次：立即执行、仅准备执行包，还是暂不执行。明确只要资产时不要重复询问。
3. 选择执行准备时，为每个 case 和运行序号初始化 `runs/{run-id}/run.json` 与 `results.csv`；选择暂不执行时不要创建 `runs/`。
4. 选择立即执行时，由 Evaluation Runner 调用产品真实入口，将真实输出、状态、耗时以及运行时可提供的 Token 和费用写入 `results.csv`。
5. 仅对执行成功且输出非空的记录，由当前调用本 Skill 的 Agent 按 `05_LLM-as-Judge评审方案.md` 评分，并回填 Judge 结果、否决候选和复核原因。
6. 有复核任务时生成 `human_review.csv`。人工完成强制复核和普通样本分层抽样复核，填写最终否决确认和最终结论。
7. 强制复核和抽样复核完成率均为 100%、未决否决候选为 0 后，才可要求正式评测报告。
8. 将值得沉淀的失败案例写入 `07_bad_case_log.csv`，后续把代表性 Bad Case 补充进 Benchmark，作为回归用例。

# 长期持续评测机制

## 目标

在产品持续迭代中，防止质量漂移、风险回潮和旧问题复发。

## 流程

1. 设定变更触发：模型、Prompt、检索、工具、输入预处理、策略、交互流程等变化。
2. 设定定期触发：每周、每月或按业务节奏运行快速或完整 Benchmark。
3. 维护 Benchmark 版本：记录新增、删除、修改和来源。
4. 维护 Rubric 版本：记录评分标准变化和校准结果。
5. 每次触发使用独立 `run-id`，由 Evaluation Runner 调用已确认的产品入口，不覆盖历史成功运行。
6. 由当前调用本 Skill 的 Agent 或用户配置的其他 Judge 方式执行 Judge，并保留 Judge 版本信息。
7. 维护 Bad Case 台账：持续沉淀失败案例。
8. 设置告警阈值：一票否决、P0/P1 通过率、幻觉率、人工接管率、延迟、成本等。
9. 明确角色分工：产品、工程 / 算法、领域专家、安全 / 法务 / 合规。
10. 定期复盘评测结果。
11. 把真实 Bad Case 补充进 Benchmark。
12. 留下决策记录。

## 变更触发

每次重要变更后，至少运行快速回归 Benchmark；高风险产品应运行高风险用例和一票否决用例。

## 定期触发

定期运行完整或抽样 Benchmark，结合线上指标、用户反馈和 Bad Case 台账做趋势分析。

## 决策记录

每次评测后记录：版本、评测范围、数据来源、关键结论、风险、决策人、决策动作和后续计划。

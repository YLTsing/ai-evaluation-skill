# Bad Case 闭环流程

## 目标

把失败案例转化为产品修复、Benchmark 回归用例和长期质量资产。

## 流程

1. 收集失败信号：`results.csv` 中的候选或 Judge 结果、`human_review.csv` 中的人工裁决、用户反馈、客服工单、线上监控、A/B 测试或复盘导入。
2. 判断是否值得进入 Bad Case 台账。
3. 写入 `07_bad_case_log.csv`。
4. 归类失败类型。
5. 判断严重度。
6. 判断是否一票否决。
7. 定位根因层：输入预处理、检索 / 知识、Prompt / 策略、模型、工具调用、产品交互、评测标准。
8. 提出修复建议。
9. 设计回归用例。
10. 判断是否加入 Benchmark。
11. 补充 Benchmark。
12. 修复后复测。
13. 更新复测结果和状态。
14. 基于台账生成阶段性 Bad Case 分析报告。

## 应进入台账的案例

- P0/P1 问题。
- 触发一票否决的问题。
- 高风险问题。
- 反复出现的问题。
- 线上真实用户反馈的问题。
- 影响核心任务完成的问题。
- 能代表一类典型失败模式的问题。
- 需要补充进 Benchmark 的问题。

## 产物关系

- Bad Case 台账是主数据源。
- Bad Case 分析报告是阶段性汇总产物。
- 默认不要一条 Bad Case 一个 Markdown 文件。
- 只有重大案例、P0 事故、合规风险案例或用户明确要求时，才单独生成单条复盘文档。

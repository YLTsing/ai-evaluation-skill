# 08_持续评测机制

## 持续评测目标

- 防止质量漂移。
- 防止已修复问题复发。
- 在模型、Prompt、检索、工具或产品流程变化后快速识别风险。
- 为上线、扩量、回滚和版本选择留下决策记录。

## 变更触发条件

以下变化应触发评测：

- 模型更新。
- Prompt 修改。
- 检索更新。
- 工具变更。
- 输入预处理变更。
- 策略变更。
- 交互流程变更。
- 重要业务规则变化。

## 定期触发节奏

- 快速迭代产品：每周运行快速回归 Benchmark。
- 稳定产品：每月运行完整或抽样 Benchmark。
- 高风险产品：每次版本变更后必须运行高风险和一票否决用例。

## 快速回归 Benchmark

- 代表性 P0/P1 用例。
- 最近新增 Bad Case 回归用例。
- 高风险场景。
- 核心任务最小覆盖。
- 由 Evaluation Runner 使用独立 `run-id` 执行，不覆盖历史成功结果。

## 完整 Benchmark

- 正常场景。
- 边界场景。
- 风险场景。
- 已知易失败场景。
- 回归场景。
- 候选执行方式：`command / http / import`。
- Judge 执行方式：默认由当前调用本 Skill 的 Agent 执行，或填写其他已确认方式。

## 运行记录管理

- 每次触发创建独立 `runs/{run-id}/run.json` 和 `results.csv`。
- 有复核任务时才生成 `human_review.csv`。
- 记录 Benchmark、Rubric、产品和 Judge 版本。
- 明确运行记录保留周期和清理责任人，不得删除仍作为决策依据的记录。

## Bad Case 更新机制

- 从离线评测、人工复核、用户反馈、客服工单、线上监控和 A/B 测试中收集失败信号。
- 符合条件的案例写入 `07_bad_case_log.csv`。
- 代表性案例补充进 `03_benchmark_cases.csv`。
- 修复后更新复测结果和状态。

## Rubric 版本管理

- 记录 Rubric 版本号、修改原因和影响范围。
- 重大修改后需要重新校准 Judge 与人工评分一致性。
- 历史评测报告应记录使用的 Rubric 版本。

## 告警阈值

| 指标 | 告警阈值 | 动作 |
| --- | --- | --- |
| 一票否决触发率 | 高于 0 或超过业务阈值 | 暂停扩量并人工复核 |
| P0/P1 用例通过率 | 低于门槛 | 暂缓上线 |
| 高严重度错误率 | 超过阈值 | 进入修复和复测 |
| 人工接管率 | 异常升高 | 分析根因 |
| 延迟 / 成本 | 超过护栏 | 评估体验和成本风险 |

## 角色分工

| 角色 | 职责 |
| --- | --- |
| 产品 | 评测目标、准入门槛、优先级和决策记录 |
| 工程 / 算法 | 执行评测、修复、自动化和监控 |
| 领域专家 | 设计风险场景、人工复核和争议仲裁 |
| 安全 / 法务 / 合规 | 一票否决规则和高风险复核 |

## 定期评审会议

- 评审周期：
- 参与角色：
- 输入材料：Benchmark 结果、Bad Case 台账、线上指标、A/B 数据。
- 输出材料：修复计划、Benchmark 更新、上线或扩量决策。

## 决策记录

每次重要变更都应记录：改了什么、测了什么、通过了什么、失败了什么、谁做了什么决策、下一步是什么。

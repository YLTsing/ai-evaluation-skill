# 评测方法论

## 范围

这份参考用于设计专业可信的 AI 产品评测体系，覆盖上线前离线评测、上线后验证、Bad Case 闭环和长期持续评测机制。

## 1. 上线前离线评测

离线评测回答的问题是：“这个 AI 产品是否足够好，可以进入真实用户验证？”

### 1.1 定义评测指标

评测指标用于在打分之前先定义什么是“好”。

从三个通用维度出发：

| 通用维度 | 核心问题 |
| --- | --- |
| 准确性 | 输出是否事实正确、逻辑正确？ |
| 相关性 | 输出是否真正回答了用户任务？ |
| 用户满意度 / 任务完成度 | 用户能否基于输出完成想做的工作？ |

再扩展到场景专属维度：

| 扩展问题 | 作用 |
| --- | --- |
| 这个维度里有哪些不同失败模式？ | 把模糊维度拆成可以独立判断的项目。 |
| 除了直接用户，还有谁会被输出影响？ | 覆盖次级用户、被评价对象、运营人员、审核人员、下游团队等。 |
| 哪些领域风险必须显性化？ | 明确安全、公平性、合规、隐私、财务、医疗、法律、品牌等风险。 |

当某个失败无论平均分多高都必须阻断上线时，把它设为一票否决维度。

### 1.2 设计 Benchmark

Benchmark 是上线前设计的一组固定测试用例。它要覆盖重要场景，而不是随机抽样。

从评测维度反推 Benchmark：

- 每个核心维度至少对应一类用例。
- 每类用例说明输入形态、预期行为和典型失败模式。
- 同时包含正常场景、边界场景、风险场景和已知易失败场景。
- 保持 Benchmark 稳定，便于跨版本比较。
- 上线后把真实 Bad Case 补充进去，让 Benchmark 越来越接近真实世界。

Benchmark 用例定义与运行结果应分开维护：

- `03_benchmark_cases.csv` 只保存稳定的用例、风险和规则。
- 每个产品版本和运行配置使用独立 `run-id`。
- `run.json` 保存整轮共享配置，`results.csv` 保存候选执行和 Judge 结果。
- 人工复核与最终裁决保存在 `human_review.csv`，避免与 Judge 的否决候选混淆。
- 待执行、失败、空输出和仅准备状态不得计入真实评测结果。

当真实输入较长、包含多文件、需要保留排版/代码/对话结构、使用二进制文件或需要独立版本冻结时，将产品实际输入保存到 `benchmark_materials/`，在 `输入` 或 `上下文/素材` 中保留引用。短且清晰的内容继续内联。素材包不是评测答案：候选产品只能看到解析后的任务和素材；预期行为、失败模式、Rubric 和风险规则只提供给 Judge 与人工评审。

### 1.3 编写 Rubric

Rubric 是人工评审或 LLM-as-Judge 使用的评分标准。

对每个维度：

- 定义分数等级，常见是 1-3 分或 1-5 分。
- 先定义最高分和最低分，再推出中间分。
- 每个分数等级给出具体示例。
- 一票否决检查清单要与普通分数等级分开。

Rubric 的目标不是文采，而是降低评分自由度，让不同评审者能打出接近的分数。

### 1.4 使用 LLM-as-Judge 并进行人工校准

LLM-as-Judge 可以扩大评测规模，但必须被约束：

- 给评审模型提供用户输入、上下文、产品输出、指标定义、Rubric 和评分格式。
- 要求逐维度打分，并给出简短、基于证据的理由。
- 当疑似触发一票否决规则时，Judge 只能标记 veto candidate；最终否决由人工复核确认。
- 条件允许时，避免评审模型和被评测模型使用同一模型或同一厂商模型。
- 一票否决、异常分数和抽样样本需要升级到人工复核。
- 定期比较人工分数和模型评审分数，校准自动评测可信度。

### 1.5 判断上线准入

常见上线前门槛：

- 没有触发一票否决。
- 每个核心维度平均分达到最低阈值。
- 高优先级用例类别单独达标，而不是只看总平均分。
- 已知高严重度问题已有修复，或有明确上线限制。

如果评测不达标，先判断问题出在产品本身、Benchmark、Rubric，还是评测指标定义，再决定是否修改产品。

## 2. 版本对比与 A/B 测试

AI 版本对比必须控制随机性。

### 2.1 离线对比

进入线上流量前：

- 用同一套 Benchmark 跑每个版本。
- 多轮运行并取平均值。
- 比较逐维度、逐用例类别结果，而不是只看总分。
- 检查分数波动和一票否决触发。

只有当离线结果足够稳定，才值得进入线上验证。

### 2.2 在线 A/B 测试

在线 A/B 测试回答的问题是：“这个变化在真实生产场景里是否提升了用户价值？”

需要定义：

- 对照组：现有工作流、基线模型或人工流程。
- 实验组：模型、Prompt、检索、交互或工作流版本。
- 分流单位：选择不会互相污染的单位，例如账号、任务、项目、工作区、业务对象。
- 主指标：业务价值或任务价值指标。
- 护栏指标：安全、公平性、幻觉、延迟、成本、用户推翻率、投诉率、高严重度错误率。
- 实验周期和扩量计划。
- 胜出、失败、不确定、回滚或扩大灰度的判定规则。

不要因为一次离线分数差异，就直接判定某个版本胜出。

## 3. Bad Case 分析

Bad Case 分析把失败案例转化成产品改进和 Benchmark 更新。

### 3.1 收集失败信号

同时收集主动信号和被动信号：

| 信号类型 | 示例 |
| --- | --- |
| 主动信号 | 用户反馈、人工复核拒绝、客服工单、专家标注。 |
| 被动信号 | 反复重新生成、用户推翻 AI 结论、流程放弃、延迟异常、指标突然漂移、人工修正率升高。 |

### 3.2 归类和排序

按失败类型和严重度归类。

一票否决类问题即使频率低，也应优先处理。非一票否决的可用性和质量问题，再更多考虑频率和影响面。

### 3.3 定位根因层

提出修复前，先定位问题发生在哪一层：

| 根因层 | 典型失败 |
| --- | --- |
| 输入预处理 | 来源内容未解析、表格丢失、OCR 失败、元信息缺失。 |
| 检索 / 知识 | 来源缺失、来源过期、上下文不相关、排序失败。 |
| Prompt / 策略 | 指令模糊、缺少判断规则、约束冲突。 |
| 模型 | 推理失败、格式不稳定、领域能力不足。 |
| 工具调用 | API 调错、工具结果缺失、中间状态错误。 |
| 产品交互 | 用户无法纠正、无法查看证据、无法理解不确定性。 |
| 评测标准 | 指标模糊、Rubric 不一致、Benchmark 缺少场景。 |

### 3.4 修复和验证

每个修复都要：

- 定义预期行为。
- 跑之前失败的定向用例。
- 跑邻近场景回归用例，确认没有破坏原本正常的能力。
- 如果这个失败来自真实生产场景，至少补充一个代表性用例进 Benchmark。
- 记录结果、负责人和后续动作。

## 4. 长期持续评测机制

持续评测用于防止质量漂移和旧问题复发。

### 4.1 触发方式

使用两类触发：

- 变更触发：模型更新、Prompt 修改、检索更新、工具变更、输入预处理变更、策略变更、交互流程变更。
- 定期触发：按固定周期跑完整或抽样 Benchmark，例如快速迭代系统每周一次，稳定系统每月一次。

### 4.2 角色分工

根据组织情况调整角色：

| 角色 | 职责 |
| --- | --- |
| 产品 | 评测目标、质量门槛、上线阈值、优先级排序、决策记录。 |
| 领域专家 | 判断标准、标注样本、争议复核、风险场景设计。 |
| 工程 / 算法 | 执行评测、修复问题、自动化、监控、回归测试。 |
| 安全 / 法务 / 合规 | 一票否决规则、敏感维度、监管风险的定期审查。 |

### 4.3 运转节奏

持续维护：

- Benchmark 变更记录。
- Rubric 版本记录。
- 评测运行记录。
- Bad Case 台账。
- 告警阈值。
- 定期评审节奏。

每次重要产品变更都应该留下评测记录，说明改了什么、测了什么、通过了什么、失败了什么、最终做了什么决策。
